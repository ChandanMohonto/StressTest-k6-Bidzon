Detected Bottlenecks

🔴 Socket.IO Connection Pool Saturation

At 2,000 concurrent users per machine (12,000 total), Socket.IO polling may saturate Node.js connection limits. Monitor PM2 active handles. Increase UV_THREADPOOL_SIZE if p99 spikes above 2s at peak.

🟡 PostgreSQL Connection Pool Pressure

Each bid triggers a DB write. Knex default pool of 10 will queue aggressively under 12K concurrent users. Watch for timeout errors during peak and spike phases.

🟡 Concurrent Auction Lifecycle Contention

100 user VUs per machine simultaneously create, update, pause, resume, and close auctions, causing write contention on the auctions table. Watch for DB lock wait increases.

🔵 Auction Timer Broadcast Storm

Server emits auction:timer every second to all room members. A single room with 2,000 users generates 2,000 msgs/sec — the highest risk event loop blocker at this scale.

Recommendations
1

Increase PostgreSQL pool — pool.max=100 + PgBouncer

Set Knex pool.max=100 and add PgBouncer as a connection pooler to handle concurrent bid writes from 12K users.

2

Run PM2 in cluster mode with Redis adapter

pm2 start app.js -i max across all CPU cores. Add @socket.io/redis-adapter so auction rooms work across instances.

3

Cache GET /api/auction/auctions with Redis (5s TTL)

Every VU calls this on login. A short Redis cache reduces DB reads by ~95% at peak with zero user impact.

4

Throttle auction:timer to every 3s for large rooms

Reduces event loop broadcast pressure by 66% for rooms with more than 500 concurrent bidders.

5

Queue auction lifecycle operations via BullMQ

Route create/update/close through a job queue to prevent DB write contention under concurrent user load.
